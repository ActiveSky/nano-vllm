# Nano-vLLM 中文导读

这份导读的目标很简单：帮助你用一条清晰的主线读懂这个项目，而不是把所有文件平铺开来硬啃。

Nano-vLLM 是一个从零实现的轻量级推理引擎，核心目标是尽量用较少的 Python 代码复现 vLLM 的几项关键能力：批量调度、KV cache、prefix cache、tensor parallel、CUDA graph 和采样生成。

## 一次生成是怎么跑起来的

你可以把一次生成流程理解成下面这条链：

`prompt -> tokenizer -> Sequence -> Scheduler -> ModelRunner -> Qwen3 -> Sampler -> output`

更具体一点：

1. `example.py` 里先创建 `LLM`，再调用 `generate()`。
2. `LLM.generate()` 把每个 prompt 包装成 [Sequence](nanovllm/engine/sequence.py) 对象，并交给 [Scheduler](nanovllm/engine/scheduler.py)。
3. `Scheduler.schedule()` 先尝试 prefill，如果没有新的 prompt，就进入 decode。
4. [ModelRunner](nanovllm/engine/model_runner.py) 负责把输入整理成张量，调用模型前向，并在需要时使用 CUDA graph。
5. [Qwen3ForCausalLM](nanovllm/models/qwen3.py) 执行真正的模型结构。
6. [Attention](nanovllm/layers/attention.py) 会读取全局 [Context](nanovllm/utils/context.py)，决定当前是 prefill 还是 decode，并写入 KV cache。
7. [Sampler](nanovllm/layers/sampler.py) 把 logits 转成最终 token。
8. `Scheduler.postprocess()` 把 token 写回序列，判断是否结束，并在结束时释放块。

## 你需要先理解的几个核心概念

### 1. Sequence

[nanovllm/engine/sequence.py](nanovllm/engine/sequence.py) 负责描述一条请求的完整状态。它不仅保存 token_ids，还保存：

- prompt 和 completion 的边界
- 当前状态是 waiting、running 还是 finished
- 已缓存 token 数
- block_table，也就是这条序列占用了哪些物理块

如果你想知道“一个请求从进入系统到结束，中间状态怎么变化”，先看这个文件。

### 2. BlockManager

[nanovllm/engine/block_manager.py](nanovllm/engine/block_manager.py) 负责 KV cache 的物理块管理。

它做的事可以概括成三件：

- 给序列分配块
- 释放块
- 用哈希判断前缀是否可以复用

这就是这个项目里 prefix cache 的核心。

### 3. Scheduler

[nanovllm/engine/scheduler.py](nanovllm/engine/scheduler.py) 决定“这一轮到底跑哪些序列”。

它会在两种模式之间切换：

- prefill：处理新请求的 prompt
- decode：对已在运行中的序列逐 token 生成

如果空闲块不足，它还会做 preemption，把部分序列暂时赶回 waiting 队列。

### 4. ModelRunner

[nanovllm/engine/model_runner.py](nanovllm/engine/model_runner.py) 是真正执行模型的地方。

这里包含几个重要动作：

- 初始化分布式进程组
- 加载模型权重
- 分配 KV cache
- 预热模型
- 在部分场景下捕获 CUDA graph
- 组织 prefill / decode 的输入张量

如果你想理解“为什么这个项目能快”，这一层很关键。

### 5. Qwen3 模型和 layers

[nanovllm/models/qwen3.py](nanovllm/models/qwen3.py) 是模型结构本体，它把各个算子拼成标准的 decoder-only Transformer。

其中 [nanovllm/layers/](nanovllm/layers) 目录下的文件分别负责：

- `linear.py`：张量并行线性层
- `embed_head.py`：词表并行 embedding 和输出头
- `attention.py`：注意力和 KV cache 写入
- `layernorm.py`：RMSNorm
- `rotary_embedding.py`：RoPE
- `activation.py`：SiLU + Mul
- `sampler.py`：采样

你可以把这些文件理解成“构建模型的积木”。

## 目录级别的阅读建议

### 根目录

- [example.py](example.py)：最小使用示例，适合从这里开始。
- [bench.py](bench.py)：性能测试脚本，适合看完整调用链但不适合第一次阅读。

- [nanovllm/config.py](nanovllm/config.py)：启动配置。

### engine

- [sequence.py](nanovllm/engine/sequence.py)：请求状态。
- [block_manager.py](nanovllm/engine/block_manager.py)：块管理和 prefix cache。
- [scheduler.py](nanovllm/engine/scheduler.py)：调度策略。
- [model_runner.py](nanovllm/engine/model_runner.py)：模型执行器。
- [llm_engine.py](nanovllm/engine/llm_engine.py)：总入口，负责把前面几层串起来。

### layers 和 models

- [nanovllm/models/qwen3.py](nanovllm/models/qwen3.py)：模型结构。
- [nanovllm/layers/](nanovllm/layers)：算子实现。

### utils

- [nanovllm/utils/context.py](nanovllm/utils/context.py)：线程/进程内共享上下文。
- [nanovllm/utils/loader.py](nanovllm/utils/loader.py)：权重加载。

## 读代码时要记住的几个词

- prefill：处理 prompt 的阶段。
- decode：逐 token 生成的阶段。
- block：KV cache 的物理分块。
- prefix cache：复用前缀块，减少重复计算。
- tensor parallel：按维度切分参数，在多卡上并行计算。
- CUDA graph：用固定图减少 kernel launch 开销。

## 最后一个建议

如果你只想抓住主线，不要一开始就钻进每个算子。先按下面这条路线读：

`example.py -> llm_engine.py -> scheduler.py -> model_runner.py -> qwen3.py -> layers/`

读通这条线以后，再回头看 [block_manager.py](nanovllm/engine/block_manager.py) 和 [context.py](nanovllm/utils/context.py)，你会更容易理解为什么注意力、缓存和调度能够协同工作。
